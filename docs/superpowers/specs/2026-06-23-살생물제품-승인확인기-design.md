# 약국 제품 살생물제품 승인 확인기 — 설계 문서

작성일: 2026-06-23

## 1. 목적

**2026년 7월 1일 살생물제 승인제 전면 시행** → 승인 안 받은(미승인) 살충제·살생물 용도 제품은 판매 금지.
약국이 취급하는 제품(살충제·살균제·기피제 등)이 환경부 화학제품관리시스템(chempold)의
**"승인 살생물제품"** 목록(=판매 가능 화이트리스트)에 있는지를 **폰에서 즉시** 확인해 재고를 분류하는 도구.

- 입력: 제품명(직접입력+드롭다운) (1단계), 또는 바코드 스캔 (2단계)
- 출력:
  - 목록에 있음 → ✅ **승인됨 · 판매 가능** + 제품 상세 카드
  - 목록에 없음 → ⚠️ **미승인 · 7/1부터 판매 불가 대상**(살충제·살생물 용도 제품 한정) + 공식 공지·제조사 확인 권고
- 핵심: 같은 브랜드라도 SKU별로 승인 여부가 갈리므로(예: 에프킬라 일부만 승인) **정확한 제품명 확인이 관건**.
- 검증 사례(2026-06-23 기준): 비오킬·신기패 = 목록에 없음(판매불가 대상) / 에프킬라수성에어로졸 그린플로랄·시트러스, 해피홈파워리퀴드에스액 = 목록에 있음(판매 가능).

## 2. 데이터 출처

- 목록 페이지: `https://chempold.keiti.re.kr/cpms/bcp/disc/bcp000List.do`
  - "승인 살생물제품 정보공개", 현재 총 **244건** (변동 가능)
  - 서버사이드 렌더링(eGovFrame). POST 폼 `name="search"`로 결과 HTML 반환.
  - 인코딩: **UTF-8** (헤더·meta 모두 UTF-8)
  - 검색 필드: `searchKey1` (SC001=전체 / SC002=살생물제품 명칭 / SC003=살생물물질 명칭),
    `searchKey2` (검색어), `countPerPage`(페이지당 건수), `pageIndex`
  - 전체 수집 방법: `countPerPage`를 크게 잡거나 `pageIndex`를 1~끝까지 순회하여 전 건 수집
- 목록 컬럼(8): 순번 / 정보공개번호(bcpDiscSn) / 살생물제품 명칭 / 살생물물질 명칭(CAS No. 함량) /
  사용가능 살생물제품 유형 및 기능 / 사용자 범위 / 승인 유효기간 / 승인정보
- 승인정보 값: "상세보기"(공개 완료) 또는 "작업중"(상세 준비중이나 **승인은 된 상태**)
- 상세 페이지: `https://chempold.keiti.re.kr/cpms/bcp/disc/BcpDisclosureView.do?bcpDiscSn=<정보공개번호>`
  (GET으로 바로 열림. "작업중" 제품은 아직 미게시)

## 3. 전체 구조

두 개의 조각으로 구성한다.

### (A) 업데이트 스크립트 — PC에서 가끔 실행 (Python)

역할: chempold에서 전체 데이터를 수집 → 정제 → 검색 페이지에 데이터를 주입.

1. chempold 목록에 POST하여 전 건(244건+) 수집
2. EUC-KR 디코딩 후 HTML 테이블 파싱 → 각 제품을 구조화된 레코드로 변환
   - `정보공개번호, 제품명, 물질명_CAS, 제품유형, 사용자범위, 유효기간, 상세상태(상세보기/작업중)`
3. **상세 페이지 수집:** 상태가 "상세보기"인 제품은 `BcpDisclosureView.do?bcpDiscSn=...`도 GET 수집·파싱하여
   상세 항목(제품 정보/살생물물질/사용방법/주의사항 등)을 구조화해 함께 저장 → **상세도 오프라인 열람**
   ("작업중" 제품은 상세 없음 → "준비중"으로 표기)
4. 정규화 키 생성: 제품명을 소문자화·공백제거한 `검색키` 필드 추가(부분일치/공백무시 검색용)
5. 산출물:
   - `data/products.json` (수집 데이터 + 기준일 + 총건수)
   - 이 JSON을 묻혀 넣은 **자체 완결형 HTML 1개** 생성 (`살생물제품검색.html`)
   - HTML 상단에 "기준일: YYYY-MM-DD · 총 NNN건" 표시
5. 날짜는 스크립트 실행 시 인자/시스템에서 주입(워크플로 제약과 무관, 일반 실행 스크립트임)

### (B) 검색 페이지 — 폰에서 사용 (HTML 1개 파일)

- 모바일 우선 레이아웃, **탭 구조**:
  - **[검색]** 탭: 제품명 키워드 검색
  - **[전체 목록]** 탭: 바코드/검색 없이 244건을 그냥 둘러보기(스크롤·간단 필터)
  - **[스캔]** 탭: 카메라 바코드 (2단계에서 추가)
- 데이터(목록+상세)가 파일 안에 포함 → **인터넷 없이 검색·상세 모두 동작**
- 상세보기는 우리 화면에서 가독성 있게 렌더링(원문 링크도 함께 제공)

## 4. 검색 동작 (1단계) — 콤보박스(직접입력 + 드롭다운 + 근거기록)

- **검색창 = 자유 타이핑(직접입력)**. 대충 쳐도 됨. placeholder 예시("예: 홈키파, 에프킬라").
- 입력하는 동안 **일치 후보가 드롭다운**으로 표시(부분 일치 + 공백 무시 + 대소문자 무시).
  - 검색 색인 = **제품명 + 살생물물질명 + 추가제품명**(상세에서 수집). "홈키파"는 추가제품명에 있으므로 이렇게 해야 잡힘.
  - 예: "홈매트" → "홈매트 리퀴드 에스" 등, "홈키파" → 홈매트 계열 매칭.
- **끝내는 방법 두 가지:**
  1. **드롭다운에서 제품 선택** → ✅ 승인됨·판매 가능 + 결과 카드(+상세)
  2. **드롭다운 맨 아래 "없음" 선택** → ⚠️ 미승인(판매 불가 대상). 단 **익명 불가**: 그 순간 입력한 검색어가 근거로 함께 처리됨.
- **"없음"은 근거를 항상 동반:** 빈 입력이면 "없음" 불가. 저장 전 되묻기("'OOO'로 검색했는데 목록에 없습니다. 처리할까요?").
  1단계에선 화면에 "'OOO'은 목록에 없음(해당없음)"으로 입력어를 그대로 echo. (2단계에선 이 검색어·날짜가 ✕ 캐시 근거로 저장 → §5)
- **[전체 목록] 탭:** 검색 없이 244건 전체를 카드/행으로 둘러보기. 제품유형(살충제/살균제 등) 간단 필터 제공.

### 결과 카드 표시 항목 (6 + 상태)

1. 살생물제품 명칭
2. 살생물물질 명칭(CAS No. 함량)
3. 사용가능 살생물제품 유형 및 기능 (예: 살충제)
4. 사용자 범위
5. 승인 유효기간
6. 정보공개번호
- **승인상태 배지**: "상세보기" → 버튼 활성화(아래 상세 화면 열림) / "작업중" → 비활성 + "상세 준비중" 표기.
  단, 작업중이어도 카드 자체는 **✅ 해당됨**으로 취급(유효기간 존재).
- **가독성:** 바코드 스캔/검색 결과 카드는 한눈에 ✅/❌가 보이도록 큰 상태 배지 + 핵심 정보 우선 배치.

### 상세 화면 (가독성 있게 정리)

- "상세보기" 클릭 시, 업데이트 스크립트가 미리 수집한 상세 데이터를 **우리 레이아웃으로 깔끔하게 렌더링**
  (항목을 섹션으로 그룹화: 제품 기본정보 / 살생물물질·함량 / 사용 방법·범위 / 주의사항 등)
- 하단에 "공식 원문 보기" 링크(온라인) 함께 제공
- "작업중" 제품은 상세 데이터 없음 → "상세 준비중" 안내

## 4-1. 검색 정확도 · 결과 해석 (중요)

- **완전성:** 승인 제품 전 건(244+)을 통째로 수집하므로, "명단에 없음 = 해당 안됨"이 성립한다.
  허가 아닌 제품을 따로 열거할 필요 없음(화이트리스트 방식, 자동 100% 커버).
- **주된 위험 = 표기 차이로 인한 거짓 "해당 안됨":** 허가됐으나 검색어 표기가 달라 못 찾는 경우.
  - 완화: 공백무시·부분일치·대소문자무시 검색, 물질명/CAS 보조 검색
  - 결과 없을 때 단정 금지 → "목록에서 못 찾음. 띄어쓰기·오타 확인 또는 공식 사이트 재확인" 안내
- **"미승인" 표시의 의미 한정:** "이 승인 살생물제품 목록에 없음"이며, 살충제·살생물 용도 제품이면 7/1부터 판매 불가 대상.
  단 행정 처리 지연·미정 사례(예: 일부 브랜드)도 있으니 단정 금지 → **공식 공지·제조사 확인** 권고를 화면에 명시.
  (비-살생물 용도 제품이라면 애초에 이 규제 대상이 아닐 수 있음.)
- **범위 확정:** 안전확인대상생활화학제품(초록누리, 별도 포털)은 이번 살생물제 승인과 별개이며 수집 제외.
  과거 안전확인 승인이 있어도 살생물제 승인목록에 없으면 7/1부터 판매 불가 대상(예: 신기패).

## 5. 바코드 (2단계, 1단계 완료 후 추가)

전제: chempold·재고프로그램 어디에도 바코드↔제품 매핑이 충분치 않음 → **자동 인식 데이터는 처음엔 없음.**
따라서 이 도구가 매핑을 **직접 쌓아가는** 방식으로 설계한다.

### 바코드 메모장 = 3가지 상태

각 바코드는 로컬에 다음 중 하나로 기록된다(`localStorage`).

- **연결됨(○)**: `바코드 → 정보공개번호`. 244목록 실제 제품에 묶임(안전).
- **허가없음으로 확인됨(✕)**: `바코드 → {결과:없음, 검색어, 확인일}`. 사람이 검색 후 "목록에 없음" 확인.
- **모름(?)**: 기록 없음.

- 동작
  1. 카메라로 바코드 스캔(EAN-13 등). 웹 카메라 라이브러리 사용(iOS/Android 브라우저 지원).
  2. 기록이 **연결됨(○)** → 즉시 ✅ 해당됨 + 카드
  3. 기록이 **허가없음(✕)** → 즉시 ❌ 해당없음 (단, "○○○○-○○-○○ 확인됨 · 다시 확인하기" 버튼 노출)
  4. 기록이 **없음(?)** → 제품명 검색 유도:
     - 검색해서 나오면 → ✅ + 바코드를 **연결됨(○)** 으로 저장
     - 검색해도 없으면 → 확인 후 **허가없음(✕)** 으로 저장(검색어·날짜 함께)
  5. 바코드 인식 실패/애매 → 제품명 타이핑 검색으로 폴백(1단계 그대로)

### 허가없음(✕) 캐시 안전장치

- "없음" 선택 시 **입력한 제품명이 곧 근거** → ✕ 기록에 **검색어 + 확인일** 동봉(익명 저장 불가).
- 저장 전 되묻기: "'OOO'로 검색했는데 목록에 없습니다. '허가없음'으로 저장할까요?"
- **신뢰 표시:** 같은 바코드 재스캔 시 화면에 `❌ 허가없음 · 'OOO'로 YYYY-MM-DD 확인됨` 처럼 **누가 무슨 단어로 언제** 확인했는지 노출 → 다음 사람이 근거를 보고 판단.
- **다시 확인하기:** 누르면 **저장된 검색어가 검색창에 미리 채워진 채** 재검색 → 허술한 근거를 한 번에 정정.
- **자동 재검증:** 데이터 갱신 시, ✕의 검색어가 새 목록에 매칭되면 그 ✕를 자동 해제(낡은 기록 정리).
- 모든 기록(○/✕) **수정·삭제 가능**, 잘못된 연결은 본인이 고치고 그 수정이 그대로 반영·유지

- 매핑 저장: 브라우저 `localStorage`(폰 내부). **백업/내보내기·가져오기(JSON)** 포함 → 기기 변경·공유 시 기록 이전.
- 기술 제약: **카메라는 보안 컨텍스트(HTTPS)에서만 동작.**
  → 2단계부터는 파일을 그냥 여는 방식 대신 **무료 HTTPS 호스팅(예: GitHub Pages)**으로 전환 필요(비용 0).
  → 1단계는 로컬 파일로도 충분하나, 2단계 전환을 고려해 호스팅 옵션을 안내한다.
- 외부 바코드 자동조회 DB: **범위에서 제외**(무료 커버리지 불확실).

## 6. 배포 · 공유 (확정: GitHub Pages)

- **호스팅: GitHub Pages** (무료 HTTPS). 링크(`https://<id>.github.io/<repo>/`) 하나를 공유하면
  누구나 폰 브라우저로 열어 사용. 카메라/바코드(HTTPS 필요)도 이 위에서 동작.
- 업데이트: 업데이트 스크립트로 HTML 재생성 → 레포에 다시 올리면(push) 전원에게 즉시 반영.
- 공공 데이터라 링크 공개 무방. (계정 생성·업로드 절차는 구현 단계에서 단계별 안내)
- 보조: 인터넷 없이 단독 배포가 필요하면 같은 HTML 파일을 카톡/메일로 전달해도 검색은 동작
  (단 파일 방식은 카메라 불가).

### 접근 · 보안 (확정: 공개 링크 + 소스 비공개)

- **배포 사이트는 공개**: 링크 아는 사람은 누구나 사용. 정적 웹이라 client-side 코드/데이터는
  열람·복사 가능(기술적으로 차단 불가). 단 **데이터가 공개 정부자료 + 단순 검색 UI라 복사 실익 없음.**
- **GitHub 소스 레포는 private**: 수집(스크래핑) 스크립트·프로젝트 파일은 비공개로 노출 안 함.
  (배포된 HTML 자체는 보이지만 빌드 과정·도구는 가려짐)
- 서버·DB·로그인·개인정보 수집 없음 → 공격 표면 최소. HTTPS로 통신 암호화.
- 고유 자산인 **바코드 메모(○/✕)는 공개 사이트에 올라가지 않고 각 기기 localStorage에만** 존재.
- 비밀번호/인증 기반 접근 제한은 백엔드가 필요하므로 현재 범위 밖.

### 바코드 메모 공유 (확정: 각자 저장 + JSON 공유)

- 호스팅으로 공유되는 것은 **앱 자체**. 바코드 메모(○/✕)는 **각 기기 `localStorage`에 저장**.
- 사람 간 공유는 **JSON 내보내기/가져오기**로 수동 처리.
- "한 명 등록 → 전원 자동 반영"이 필요해지면 **공용 저장소(백엔드)** 를 얹는 별도 단계로 확장(현재 범위 밖).

### 사용 횟수 카운트 (확정: GoatCounter 익명 페이지뷰)

- 목적: "사람들이 몇 번 열어봤나"를 사장님이 대시보드에서 확인(호기심·운영 참고).
- 방식: **GoatCounter** 스크립트 1줄을 페이지에 삽입 → **앱이 열릴 때마다 익명 페이지뷰 1회** 집계.
  쿠키·개인정보 수집 없음. 무료(개인용). 대안 Cloudflare Web Analytics(동등).
- **바코드 메모(○/✕)는 전송하지 않음** — 나가는 것은 익명 "열림" 신호뿐. "내 데이터 안 나감" 약속 유지.
- 인터넷 연결 시에만 집계(오프라인 사용은 미집계 — 자연스러운 한계).
- 설정값(본인 GoatCounter 코드)은 `template.html`의 표시된 자리에 넣고 재빌드. 비우면 카운트 비활성.

## 7. 데이터 최신화

- 데이터 변동 시 PC에서 업데이트 스크립트 재실행 → HTML 재생성 → 폰으로 재전달(또는 호스팅 재배포)
- HTML에 "기준일 · 총건수" 표기로 데이터 시점 확인

## 8. 테스트

- 스크립트: 알려진 검색어("홈매트")로 수집 시 5건이 잡히는지, EUC-KR 한글이 깨지지 않는지 검증
- 파싱: 8개 컬럼이 올바른 필드로 매핑되는지, "작업중" 상태가 보존되는지
- 검색: 공백/대소문자 무시 부분일치 동작, 미존재어 → "해당 안됨"
- 표시: 작업중 제품의 상세 버튼 비활성화

## 9. 범위 밖 (YAGNI)

- 실시간 서버 조회/백엔드(이번엔 안 함)
- 외부 바코드 DB 자동조회
- 살생물물질(244건과 별개인 nbm000List) 페이지 통합 — 이번 대상은 살생물제품(bcp000List)만

## 10. 단계 정리

- **1단계(먼저 완성):** 업데이트 스크립트 + 오프라인 제품명 검색 HTML
- **2단계(이후 추가):** 카메라 바코드 스캔 + 바코드 등록·기억(localStorage), HTTPS 호스팅 전환
